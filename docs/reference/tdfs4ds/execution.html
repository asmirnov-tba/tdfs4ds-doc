    tdfs4ds.execution¶

    tdfs4ds.execution ¶

    Feature ingestion and process execution for the tdfs4ds feature store.

    Core execution call chain:

    upload_features(df, entity_id, feature_names, ...)
  → register_process_view.__wrapped__(...)
  → execute_query(query_insert / query_insert_dist / query_insert_filtermanager)
  → [if optimize_query=True and instruct model available]
      → skill_optimize_query(process_id=process_id)
          → saves full Markdown report to tdfs4ds_optim_{timestamp}_{view}_{process_id}.md
          → if improvement found and update_view=True: REPLACE VIEW {view_name} AS {best_sql}
  → [if document=True and instruct model available]
      → document_process_incremental(process_id=process_id, display=False, upload=True)
  → run(process_id)          # ingests from the (now-optimized) view; builds dataset view if dataset_view_name set
  → [if document=True and instruct model available and dataset_view_name is not None]
      → document_dataset_incremental(dataset_id)   # bottom-up dataset docs after feature docs exist
      → SELECT * FROM process_catalog WHERE PROCESS_ID = ?
      → _upload_features(df, entity_id, feature_names, feature_versions=process_id, ...)
          → Gettdtypes / update_varchar_length / validate_feature_types
          → register_entity / register_features
          → _compute_features_batch(...)  [once, or per-filter loop]
              → followup_open
              → prepare_feature_ingestion   # unpivots df into volatile table
              → store_feature               # MERGE/UPDATE_INSERT into feature tables
              → followup_close (COMPLETED)
          → apply_collect_stats(...)        # called once after all batches

roll_out(process_list, time_manager, ...)
  → iterates time steps → run(process_id) per step

    ACC_6881_HINT module-attribute ¶

    ACC_6881_HINT = "Error 6881 (not eligible for the Analytical Compute Cluster): for a tdstone2-Apply view that references a TimeManager/FilterManager, TWO things are needed — (a) the manager table on OFS (storage='OFS'), so the ACC can read it, AND (b) the filter written as a JOIN to the manager view, NOT a correlated scalar subquery '(SELECT BUSINESS_DATE FROM <view>)', which the ACC rejects. OFS + JOIN scores; OFS + scalar subquery still 6881s. The filter shape is the user's responsibility. Also: run the model's score() BEFORE upload_features — roll_out/run() cannot drive an Apply model (its ingestion CTAS also hits 6881)."

    FilterManager ¶

    A utility for managing dynamic, versioned filter sets as database-backed views.

    The FilterManager enables lightweight scenario management by storing multiple filter definitions in a hidden Teradata table and exposing a public view that dynamically switches between them by filter_id. Each row in the hidden table represents a complete filter configuration. The active configuration is controlled by updating the view definition rather than rewriting table data.

    Key Features
    • Store multiple filter states (scenarios) indexed by filter_id
    • Switch filter states instantly by updating a view
    • Optionally include time-based slicing using a BUSINESS_DATE column
    • Clone filters between managers (soft or hard clone modes)
    • Prune obsolete filters to control table size
    • Retrieve current and historical filter definitions
    Workflow Overview
    1. Create a FilterManager pointing to a target view name.
    2. Load one or more filter definitions using load_filter().
    3. Switch active filters using update(filter_id).
    4. Inspect the active filter via display() or view DDL.
    5. Optionally prune or clone filters as needed.
    How It Works Internally
    • A hidden table named <view_name>_HIDDEN stores filter definitions.
    • A Teradata view named <view_name> exposes only the active filter row.
    • Each filter automatically receives a sequential filter_id (ROW_NUMBER() ordering ensures deterministic assignment).
    • If time-based filtering is used via time_column, a BUSINESS_DATE column is added and projected in all operations.

    build_dataset ¶

    build_dataset(entity_id, selected_features, view_name, schema_name=None, comment=None, return_query=False, feature_store_time=False, join_type='INNER', grouped=False, document=False, vector_indexing=False, record_lineage=False)

    Generate a SQL query and create a Teradata view to retrieve records for the specified entity IDs based on selected features.

    Parameters:¶

    entity_id : list, dict, or str The entity IDs for which records are needed. Can be a list, dictionary, or string. - If list: contains multiple entity IDs. - If dict: keys are the entity IDs. - If str: a single entity ID.

    dict

    A dictionary where the keys are feature table names, and the values are lists of tuples (feature_id, feature_version, feature_name) specifying the features to retrieve. NOTE: feature_version may be either: - a single UUID string, or - a list of dicts like:

    str

    The name of the view to be created in the database.

    str, optional

    The name of the database schema where the view will be created. Defaults to the global tdfs4ds.SCHEMA if not provided.

    str, optional

    A comment to attach to the created view for documentation purposes. Defaults to None.

    bool, optional

    If True, the function will return the SQL query string instead of executing it. Defaults to False.

    bool, optional

    If True, the query will use a specific VALIDTIME based on tdfs4ds.FEATURE_STORE_TIME. If False, the query will use CURRENT VALIDTIME. Defaults to False.

    str, optional

    Specifies the type of join to use between features: 'INNER' (default) or 'FULL OUTER'.

    bool, optional

    If False (default), uses one sub-query per feature joined together — the classic multi-join approach. Every individual feature sub-query can be targeted by a PI-equality filter pushed from the outer WHERE, making single-entity lookups single-AMP. If True, groups features that share the same storage table and process version into a single sub-query with MAX(CASE WHEN …) pivoting, reducing the number of JOINs from N_features to N_distinct_(table, version) pairs. A HAVING clause enforces non-NULL for every feature in the group when join_type='INNER', preserving the same missing-feature semantics as the classic approach. PI push-down is preserved for INNER JOIN (outer SELECT uses A1.entity); for FULL OUTER the COALESCE in the outer SELECT prevents push-down, same as the classic approach.

    bool, optional

    If True, generate and persist business-dictionary documentation for the newly built dataset via document_dataset_incremental (LLM-powered). Requires tdfs4ds.INSTRUCT_MODEL_PROVIDER and tdfs4ds.INSTRUCT_MODEL_MODEL to be configured; silently skipped (with an info log) otherwise. Failures are non-fatal — the dataset view is always created/returned regardless. Defaults to False (opt-in). Mirrors upload_features. Leave False when the dataset is built via upload_features/run() — that path already documents it, so enabling it here would double-document.

    bool, optional

    If True (and document=True and an instruct model is configured), refresh the consumer-agent vector index via build_vector_index(force_rebuild=False) after documentation so the new dataset is searchable immediately. Non-fatal. Defaults to False (opt-in).

    bool, optional

    If True, persist the dataset view's lineage subgraph into FS_LINEAGE_GRAPH via record_process_lineage (no LLM required). Non-fatal. A standalone dataset with no upstream processes records an isolated node. Defaults to False (opt-in).

    Returns:¶

    str or tdml.DataFrame - If return_query is True, returns the generated SQL query as a string. - Otherwise, returns a Teradata DataFrame containing the results.

    Raises:¶

    ValueError If an invalid join_type is provided.

    _materialise_art ¶

    _materialise_art(art_sql)

    Execute the UAF ART materialisation statements (art_sql) in list order.

    For each statement: DROP the volatile ART it declares first (idempotent — a fresh session has none; a re-run/next-partition has a stale one), then execute the statement. All via tdml.execute_sql on the active session, so the resulting session-scoped volatile ART is visible to the ingestion CTAS that follows.

    art_sql order is authoritative and must NOT be sorted/deduped — a later statement may read an ART an earlier one created. A no-op when art_sql is falsy (the classic non-UAF path). Re-raises on statement failure so the caller's followup_close(status='FAILED,...') records it.

    _compute_features_batch ¶

    _compute_features_batch(df, entity_id, feature_names, selected_features, primary_index, partitioning, entity_null_substitute, process_id, filtermanager, dataset_view_name, dataset_already_created, grouped=False, art_sql=None)

    Execute one feature ingestion batch: followup_open → [materialise ART] → prepare → store → followup_close (COMPLETED) → optional dataset build.

    Returns (features_infos, dataset_or_None) on success. On failure: logs the error, attempts to record FAILED in the followup table, then re-raises. Callers that want to continue after a failure (e.g. the filtermanager loop) should wrap this call in their own try/except and use continue rather than re-raising.

    Does NOT call apply_collect_stats — the caller is responsible for that.

    grouped is forwarded to build_dataset — see upload_features for details.

    art_sql (UAF ART view mode only): a list of INTO VOLATILE ART statements materialised before ingestion, in this same session — so the session-scoped volatile ART is visible to the ingestion CTAS. Runs after filtermanager.update() (done by the caller), so the series-input view's scalar subquery already points at the current partition. None/empty for the classic path (a no-op).

    _resolve_feature_source ¶

    _resolve_feature_source(df, sql_query=None, target_view_name=None)

    Normalise an upload_features source to a teradataml DataFrame.

    Accepts three input shapes and returns (df, view_name):

    • a teradataml DataFrame → returned unchanged (back-compat);
    • a fully-qualified schema.view → resolved to a DataFrame over that view;
    • sql_query + target_view_name → the view is CREATED from the SQL, then resolved to a DataFrame (the created view is permanent and becomes the process view re-run by run(process_id)).

    On any recoverable problem (missing schema, view already exists, SQL/​view fails to compile or is not executable) it logs a clear, prescriptive message via logger_safe('error', …) and returns (None, None) so the caller aborts without raising a traceback (the users are non-developers).

    upload_features ¶

    upload_features(df, entity_id, feature_names, metadata={}, primary_index=None, partitioning='', filtermanager=None, entity_null_substitute={}, force_compute=True, force_varchar_length=1024, dataset_view_name=None, optimize_query=False, update_view=True, document=True, max_candidates=None, cascade_redocumentation=False, vector_indexing=False, sql_query=None, target_view_name=None, art_sql=None, art_layers=None, grouped=False)

    Upload feature data to the feature store for a specified entity. All diagnostics go through logger_safe() which respects tdfs4ds.DISPLAY_LOGS.

    The feature source can be supplied in three ways:

    1. A teradataml DataFrame (df=...) — the classic path.
    2. A fully-qualified view name (df='SCHEMA.view') — for SQL users who already have a view. The schema is mandatory; a bare view name (no schema) is rejected. The view must already exist and be executable.
    3. A SQL query (sql_query='SELECT ...') together with a mandatory target_view_name='SCHEMA.view'. The query is validated with EXPLAIN, a view is created at target_view_name (the schema is mandatory; the view must NOT already exist), and that view — which is permanent and becomes the process view re-executed by run(process_id) — is used as the feature source. When sql_query is given it takes precedence over df.

    Recoverable input problems (missing schema, target view already exists, SQL or view fails to compile / is not executable) do not raise a traceback: a clear message with a fix is logged via logger_safe('error', …) and the function returns None. Check the log when a call returns None.

    Parameters¶

    sql_query : str, optional A SQL SELECT defining the feature source. Requires target_view_name. The view is created from this query (after an EXPLAIN check) and kept. target_view_name : str, optional 'SCHEMA.view' — where to create the view for sql_query (mandatory when sql_query is given; schema required; must not already exist). art_sql : list of str, optional Opt-in UAF ART view mode. A list of solitary SQL statements that materialise a Teradata UAF operator's Analytical Result Table (ART) using EXECUTE FUNCTION ... INTO VOLATILE ART(<name>). When provided, the process is registered with PROCESS_TYPE='uaf art view' and, on every run(), these statements are executed in list order, per FilterManager partition, immediately before the ingestion CTAS (same session, so the session-scoped volatile ART is visible to the CTAS). This unlocks features whose value lives in a secondary ART layer (e.g. ARIMA fit-metrics R_SQUARE/MAPE/AIC in ARTFITMETADATA, residual tests over ARTFITRESIDUALS, TD_PACF) — which a single ingestion statement cannot reach. df/sql_query+target_view_name still define the plain feature view (the one reading the layer via TD_EXTRACT_RESULTS(ART_SPEC(..., LAYER(...)))); art_sql is stored in the process METADATA and never turned into a view (a view cannot contain INTO VOLATILE ART). Order matters — a later statement may read an ART an earlier one created. For this mode optimize_query is force-disabled and the view EXPLAIN in documentation is skipped (the ART does not exist at optimize/document time). Requires art_layers. Default: None (classic 'denormalized view' path, unchanged). art_layers : list of str, optional Required when art_sql is given: the declared LAYER(...) name(s) the feature view reads (e.g. ['ARTFITMETADATA']). Used for documentation and lineage only — it is not validated against the view DDL (the view is authoritative for what is computed). Default: None. entity_id : str or list (recommended), or dict (advanced override) Entity / primary-index key(s). Pass a string for a single-column key or a list for a composite key, e.g. 'CUSTOMER_ID' or ['CUSTOMER_ID', 'EVENT_DATE'] — this is the recommended form. You do not need to specify SQL data types: they are inferred from the source DataFrame with their exact length and character set preserved (a CHAR(14) key stays CHAR(14) in the created feature table, not truncated to CHAR(1)). Only pass a dict (e.g. {'CUSTOMER_ID': 'BIGINT', 'EVENT_DATE': 'DATE'}) in the rare case where you need to override an inferred type; the supplied type strings are then used verbatim. entity_null_substitute : dict, optional Per-column replacement values for NULLs in entity keys (e.g. {'CLI_CD': '-'}), applied via COALESCE. NULL values in an entity key column that is NOT covered here will raise a ValueError during ingestion: a NULL key produces a cartesian-product MERGE (severe skew / storage blow-up). Filter the NULL rows upstream or provide a substitute. Default: {}. optimize_query : bool, optional If True and an instruct model is configured, run the query optimizer on the registered process after ingestion. The full report is saved to a file named tdfs4ds_optim_{timestamp}_{view}_{process_id}.md in the current working directory. Only the score summary is emitted to the log. Default: False. update_view : bool, optional If True and query optimization finds an improvement, replace the process view DDL with the optimized SQL. Default: True. document : bool, optional If True and an instruct model is configured, run incremental documentation on the registered process (and after any view update). If dataset_view_name is also provided, document_dataset_incremental is called after run() so that the dataset-level documentation can propagate the already-generated feature descriptions bottom-up. Default: True. max_candidates : int, optional Maximum number of SQL rewrite candidates to generate during query optimization. Overrides tdfs4ds.QUERY_OPTIMIZER_MAX_CANDIDATES for this call only. Default: None (uses the global value). cascade_redocumentation : bool, optional If True (and document=True with an instruct model configured), re-document every downstream process/dataset that depends on this process — but only when this process's documentation actually changed. The descendant set is read from the persisted lineage graph (FS_LINEAGE_GRAPH). This propagates a changed parent meaning down to everything that consumes it, at the cost of one extra LLM pass per affected descendant; hence it is opt-in. Default: False. vector_indexing : bool, optional If True (and document=True with an instruct model configured), rebuild the vector index after documentation is written so the consumer agent's search results reflect the new/updated business descriptions immediately. Uses an additive upsert (force_rebuild=False) — fast but stale entries from deleted/renamed objects persist until a full rebuild_vector_indexing() is run. Non-fatal: a failure logs a warning without aborting the ingestion. Default: False. grouped : bool, optional Forwarded to build_dataset whenever a dataset view is built (i.e. when dataset_view_name is set or tdfs4ds.BUILD_DATASET_AT_UPLOAD is True). If False (default), uses one sub-query per feature joined together — the classic multi-join approach; every feature sub-query can be targeted by a PI-equality filter, keeping single-entity lookups single-AMP. If True, groups features sharing the same storage table and process version into one sub-query with MAX(CASE WHEN …) pivoting, reducing JOIN count from N_features to N_distinct_(table, version) pairs — recommended for wide datasets with many features from the same source table. See build_dataset for the full explanation. Default: False (unchanged behaviour). Not persisted in the process catalog — a later run(process_id) without grouped=True reverts to the classic join for any dataset rebuild.
