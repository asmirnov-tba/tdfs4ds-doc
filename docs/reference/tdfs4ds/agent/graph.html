    detect_data_domain_for_object ¶

    detect_data_domain_for_object(object_name: str, prefer_domain: Optional[str] = None) -> Optional[str]

    Return the DATA_DOMAIN that contains object_name as a feature name.

    Searches the full feature catalog (all data domains). Falls back to the process catalog (VIEW_NAME) if the feature is not found by name.

    Parameters:

    Name Type Description Default
    object_name str

    Feature or view name to locate.

    		 required
    prefer_domain Optional[str]

    When the object exists in multiple domains, return this one if it is among the matches. Used to prevent silent context switches on ambiguous feature names (e.g. "Claims" appearing in multiple feature stores).

    		 None

    Returns:

    Type Description
    Optional[str]

    Matching DATA_DOMAIN string, or None if not found.

    Cross-store federation limit (v0.3.1.148): the catalog scanned here is the federated UNION across the primary + read-only stores, but resolution returns a bare DATA_DOMAIN with NO store tag (and _with_domain only switches tdfs4ds.DATA_DOMAIN, not the active store). With DISJOINT stores a domain uniquely identifies its store, so this is correct. If two stores share a domain name, _pick returns the first match (primary rows sort first → primary wins); making resolution store-aware is the documented follow-up.

    skill_lineage ¶

    skill_lineage(question: str, object_name: str, hint_view_name: Optional[str] = None, accumulated_context: Optional[dict] = None, entity_name: Optional[str] = None, hint_table_name: Optional[str] = None) -> dict

    Return the upstream data dependencies of a feature, object, or column.

    Walks the dependency graph starting from the registered process view, and returns a structured list of upstream nodes (tables and views) with their types. Does not require LLM — the graph is deterministic.

    When object_name is not a feature/dataset/process view (e.g. a plain documented table, view, or a non-feature column), it falls back to the persisted lineage graph (FS_LINEAGE_GRAPH) to position ANY documented object. A column is first resolved to its owning object (TABLE_NAME) and that object is positioned instead — see _position_object_from_graph.

    Implements fast-path caching: queries accumulated_context_graph first (cache hit → return immediately), falls back to expensive DDL scans (cache miss → rebuild and merge results for next turn).

    Parameters:

    Name Type Description Default
    question str

    User question (used to determine response language).

    		 required
    object_name str

    Feature, object (table/view/dataset), or column name.

    		 required
    hint_view_name Optional[str]

    Pre-resolved view name from a prior turn (skips catalog lookup).

    		 None
    accumulated_context Optional[dict]

    The accumulated_context_graph from state (used for cache).

    		 None
    entity_name Optional[str]

    Entity name for disambiguation (used in cache lookups).

    		 None
    hint_table_name Optional[str]

    Owning object for a column (from a prior EXPLAIN/DEFINITION turn via resolved_column_sources) — positions that object directly.

    		 None

    Returns:

    Type Description
    dict

    Dict with object_name, process_view, dependencies (list of
    dict

    {name, type, depth}), plain_summary (LLM plain-language answer),
    dict

    _from_cache (bool), and resolved_triplet when resolution succeeded.

    skill_explain ¶

    skill_explain(question: str, object_name: str, hint_view_name: Optional[str] = None, hint_dataset_features: Optional[list] = None) -> dict

    Explain how a feature is calculated by reading its process view DDL.

    Fetches the SQL definition of the registered process view and asks the LLM to translate the logic into plain business language — no SQL jargon.

    When hint_dataset_features is provided and object_name is empty, explains all features from the current dataset context (up to 10) in a single LLM call.

    _clean_object_name ¶

    _clean_object_name(name)

    Strip markdown/quote wrappers and a trailing annotation around a name.

    The intent classifier (or user) sometimes wraps the object name in markdown backticks/quotes (`provider_z_score`) or appends a parenthetical annotation (provider_z_score (from flags)). Catalog lookups are exact, case-insensitive matches, so either form matches nothing and DEFINITION wrongly falls through to "feature not found / did you mean …".

    Strips, in order: surrounding backticks/quotes/whitespace, then ONE trailing (...) group, then surrounding wrappers again. Only the surrounding wrappers and a trailing parenthetical are removed — interior characters are never touched (feature names contain _ and digits; composite entities use commas like claim_id,region; names never contain parentheses). Falls back to the wrapper-only result if stripping the parenthetical would empty the name.

    Returns the input unchanged when falsy (None / "").
