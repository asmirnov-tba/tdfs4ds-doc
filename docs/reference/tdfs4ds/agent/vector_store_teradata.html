    tdfs4ds.agent.vector_store_teradata¶

    tdfs4ds.agent.vector_store_teradata ¶

    Teradata Enterprise Vector Store backend for the tdfs4ds business dictionary.

    Provides the same public surface as the Chroma backend (:func:tdfs4ds.agent.vector_index.build_vector_index and :func:tdfs4ds.agent.vector_index.search_vector_index) but persists the three collections as Teradata vector stores via langchain_teradata:

    • {prefix}_objects -> FS_BUSINESS_DICTIONARY_OBJECTS
    • {prefix}_sections -> FS_BUSINESS_DICTIONARY_SECTIONS
    • {prefix}_columns -> FS_BUSINESS_DICTIONARY_COLUMNS

    Embeddings are produced by :func:tdfs4ds.agent.embedding.get_vector_store_embedding which selects either teradatagenai.TeradataAI (when tdfs4ds.TERADATA_AI_API_TYPE is set) or the standard LangChain Embeddings — TeradataVectorStore accepts both at the embedding= argument.

    Indexing source (v0.3.1.131): documents are supplied to TeradataVectorStore.from_datasets/add_datasets reading from a permanent CURRENT VALIDTIME view (FS_V_BUSINESS_DICTIONARY_*, created by genai.documentation.documentation_tables_creation()) — NOT via from_documents/add_documents (LangChain Document list input). The Document-list path has a confirmed bug: it internally stages the input through teradataml.DataFrame.from_dict(..., persist=True) / DataFrame.materialize(), which takes no database argument and may materialize as a volatile table (session-scoped) or a view — either way teradataml itself documents this choice as internal/opaque ("decided and taken care by teradataml"). A volatile table is invisible to the separate Vector Store REST session, causing TDML_2412: Object ... does not exist regardless of target_database (which only scopes where the FINAL collection metadata is registered, not where the input data is read from).

    Deeper pitfall found on live VCL2 testing (v0.3.1.14x): reading from the permanent view is necessary but not sufficient on its own — the way the view is handed to from_datasets/add_datasets matters too. _from_datasets_kwargs() used to wrap the view in tdml.DataFrame.from_query("SELECT ... FROM <view>") before v0.3.1.14x. Confirmed live: DataFrame.from_query(...) sets _table_name to a freshly materialized, ephemeral ml___frmqry_v_<id> view at construction time (before teradatagenai is even called), and teradatagenai bakes whatever _table_name it receives permanently into the collection's _inputdata view DDL (confirmed via SHOW VIEW on a live collection: ... FROM "data_scientist"."ml___frmqry_v_<id>"). Once Vantage garbage-collects that ephemeral view (observed to happen almost immediately — confirmed absent from DBC.TablesV within the same session that created it), _inputdata — and collectionV_<name>, which joins through it — is permanently broken: every query touching it, from any client (similarity_search, a plain teradataml query, an unrelated MCP session), fails with Error 3807 / TDML_2412: Object ... does not exist, forever, not transiently (confirmed: 3/3 retries fail identically). The fix: data= is now passed as a plain fully-qualified "schema.view" string naming the permanent view directly (see _from_datasets_kwargs) — never a from_query() result, which guarantees no ephemeral materialization ever happens.

    No row-level filtering of data= — ever. Neither from_datasets nor add_datasets exposes any filter/predicate/where-clause parameter (checked directly against the teradatagenai signatures/docstrings). Pointing data= at a filtered view instead of the full source view was considered and rejected: since _inputdata is a live view (re-evaluated on every query, not a frozen snapshot), a filtered data= would permanently narrow what the whole collection can ever return — hiding previously-indexed rows outside the filter from search forever. Separately, _index (the real embeddings table) is keyed by TD_ID, which _inputdata assigns via ROW_NUMBER() OVER (ORDER BY <text column>) — recomputed fresh on every evaluation of whatever data= points to, so any change in row count/ordering between the _index write and a later _inputdata evaluation risks silently misaligning embeddings with the wrong row's text/metadata. Always pass the entire, unfiltered source view.

    _is_not_ready_error ¶

    _is_not_ready_error(exc: Exception) -> bool

    True when exc is the transient "Collection ... cannot be updated as it is not in a ready state" 409 — the collection is still processing a prior async operation (its own creation, or a previous update), NOT a real failure. Distinguished from a generic 409/HTML service-down error (which also contains "Response Code:") by requiring the more specific "not in a ready state" phrase; must be checked BEFORE _reraise_if_service_down's generic "response code:" branch, which would otherwise misclassify this transient, retryable condition as "service unreachable/unlicensed."

    _is_transient_staging_object_error ¶

    _is_transient_staging_object_error(exc: Exception) -> bool

    True when exc is a 400 "Object ... does not exist" from similarity_search/ask — confirmed live: the Enterprise Vector Store REST backend materializes a short-lived per-query staging view (ml___frmqry_v_<id>) to run the search, and that object can be gone (dropped/never-committed/session mismatch) by the time the search reads it back — a platform-side race, not a licensing/activation problem. Must be checked BEFORE _reraise_if_service_down's generic "response code:" branch, which otherwise misclassifies this as "service unreachable/unlicensed" (every TeradataMlException from this API embeds "Response Code:", so that check alone can't distinguish a genuine gateway failure from this per-query staging race).

    _is_transient_gateway_error ¶

    _is_transient_gateway_error(exc: Exception) -> bool

    True when exc is a bare HTTP 5xx gateway error (502/503/504) — confirmed live on a VCL2 test instance: the Vector Store REST gateway occasionally returns a raw <html>...502 Bad Gateway...</html> body (or 503/504) with no relation to any code path or configuration issue — pure transient infrastructure flakiness that resolves on retry within seconds. Distinguished from a genuine "service unreachable/unlicensed" condition only by being retryable: unlike a persistent misconfiguration (missing auth, wrong URL, licensing), a gateway 502/503/504 is by definition a transient proxy/gateway-layer failure, so retrying a few times before giving up is safe and appropriate. Checked BEFORE _reraise_if_service_down's generic "<html"/"response code:" branch so a transient gateway blip doesn't immediately surface as the terminal _SERVICE_DOWN_MSG on the very first retry-able occurrence.

    _retry_if_transient_gateway_error ¶

    _retry_if_transient_gateway_error(fn, label: str, timeout_seconds: float = 60.0, poll_interval: float = 5.0)

    Call fn(), retrying while it raises a transient gateway 5xx (see _is_transient_gateway_error). Re-raises the original exception once timeout_seconds elapses, or immediately for any other exception shape.

    A short default timeout (60s, vs. the 120s/1800s used for genuine async "not ready" waits) is deliberate — a gateway 502/503/504 either clears in a few seconds or reflects a real outage, in which case failing fast and surfacing the clear _SERVICE_DOWN_MSG (via the caller's own _reraise_if_service_down) is more useful than a long silent wait.

    _reraise_if_service_down ¶

    _reraise_if_service_down(exc: Exception)

    Inspect exc and raise a clear RuntimeError when it looks like either:

    • the Vector Store client itself is unconfigured (e.g. requests' MissingSchema/InvalidURL from a request built with a None base URL — "Invalid URL 'None/api/v2/...': No scheme supplied" — meaning set_auth_token/session auth was never set up). This is distinguished from "service down" because the fix is entirely client-side (call set_auth_token), not a licensing/activation problem on the Vantage instance.
    • the Vector Store REST gateway itself is unreachable/not enabled (any other transport-level failure, or a reachable-but-wrong endpoint returning a raw HTML/status body — e.g. a gateway 502 — instead of a JSON error).

    Returns normally (does nothing) for any other exception, leaving it to the caller — in particular for the transient "not in a ready state" 409 (see _is_not_ready_error), the transient per-query staging-object race (see _is_transient_staging_object_error), and a transient gateway 502/503/504 (see _is_transient_gateway_error), which callers should retry rather than treat as fatal, so none of the three is ever raised as service-down here.

    Shared by every call site that talks to teradatagenai/langchain_teradata directly (_lookup_store_database, build_teradata_index's from_datasets/add_datasets/status calls) so neither failure mode is ever mistaken for "store doesn't exist yet, fall back to rebuild" — which would just fail again later with a much less clear message.

    _lookup_store_info ¶

    _lookup_store_info(name: str, target_database: Optional[str] = None) -> Optional[dict]

    Return {"target_database": ..., "collection_status": ...} for a named vector store, or None if no store with that name exists.

    target_database (cross-store federation, v0.3.1.148): when given, a row matches only if its own target_database also matches — because two feature stores using the same TERADATA_VECTOR_STORE_PREFIX have identically-named collections (e.g. tdfs4ds_objects) in different databases, and a name-only match would return the wrong store's status. Default None ⇒ name-only match (unchanged for existing callers).

    Uses CollectionManager.list() — the same list-and-match pattern teradatagenai uses internally (Collection.exists, VectorStore.__init__) — so a caller can discover where a store lives before attaching, instead of assuming it matches tdfs4ds.SCHEMA. This catches the case where a store was created before a target_database fix (or a schema change) and now sits in a different database than expected.

    collection_status is the authoritative signal for whether a write actually succeeded — confirmed live to take values like 'ready', 'update_failed', 'create_failed' (see _is_failed_status). A write call (from_datasets/add_datasets) returning without an exception only means "the request was accepted" (these operations are asynchronous) — it does NOT mean the operation succeeded; only collection_status reflects the true server-side outcome.

    Raises RuntimeError (via _raise_service_unavailable) when the Vector Store service itself is unreachable/not enabled — distinguished from "store not found", which returns None normally. A transient gateway 502/503/504 (confirmed live on a VCL2 test instance — pure infrastructure flakiness, unrelated to code/config) is retried a few times first via _retry_if_transient_gateway_error before falling through to that service-down classification.

    _from_datasets_kwargs(source: str) -> dict

    Build the data/data_columns/key_columns/metadata_columns kwargs for TeradataVectorStore.from_datasets/add_datasets.

    data is passed as a plain fully-qualified "schema.view" string pointing directly at the permanent FS_V_BUSINESS_DICTIONARY_* view — NOT a tdml.DataFrame.from_query(...) result.     _safe_destroy ¶

    _safe_destroy(vs_class, name: str) -> None

    Destroy a named vector store if it already exists.

    Looks up the store's ACTUAL database via CollectionManager.list() first, rather than assuming it lives at tdfs4ds.SCHEMA — a store created before a target_database/schema change can be sitting in a different database, and attaching only at the expected database would silently miss it, leaving an orphaned duplicate once a fresh store is created elsewhere on force_rebuild.

    Calls destroy() unconditionally once existence is confirmed — do NOT gate on status() first. Collection.status() reports the async create/destroy/update OPERATION history, not current existence; a store that exists but has no reported operation history returns an empty DataFrame, which previously made this function conclude "nothing to destroy" and skip the call — leaving the real collection in place. The subsequent from_datasets() call then collided with a TeradataGenAIException: Collection ... already exists (Collection.create() does its own authoritative self.exists check), silently caused by a swallowed destroy() failure below (e.g. non-admin session — destroy() requires admin privileges per its docstring) going unnoticed because this function used to only log a warning and return.

    A destroy() failure is therefore re-raised (not swallowed): silently continuing after a failed destroy just defers the same collision to the from_datasets()/add_datasets() call with a far less clear error.

    _wait_until_collection_ready ¶

    _wait_until_collection_ready(collection, name: str, timeout_seconds: float = 1800.0, poll_interval: float = 5.0) -> None

    Poll collection.get_details() — a read-only method — every poll_interval seconds until it succeeds, treating any exception as "the collection is still processing a prior async operation" (Collection create/update/destroy are all asynchronous: the server reports a request as "accepted and in progress" well before the operation actually completes, so a write issued right after can hit the collection while it's still busy and fail with a 409 "not in a ready state").

    Deliberately read-only: unlike retrying the write itself, this probe can't create additional server-side contention while checking. Raises the last exception observed if the collection is not ready within timeout_seconds (a genuinely stuck/failed operation should surface clearly rather than hang the caller forever).

    _verify_write_succeeded ¶

    _verify_write_succeeded(name: str, source: str, timeout_seconds: float = 300.0, poll_interval: float = 5.0) -> None

    After a write call (from_datasets/add_datasets) returns without raising, confirm the operation actually completed by polling collection_status (via :func:_lookup_store_info) until it reports 'ready'.

    A write call returning cleanly only means "the request was accepted" (Collection writes are asynchronous) — NOT that the operation succeeded. Confirmed live: a store can log "indexed successfully" here while collection_status is actually update_failed/ create_failed and the collection holds no searchable data.

    If collection_status ever reports a *_failed status while polling, raises immediately via :func:_raise_collection_failed (no point waiting further — it's terminal). Raises a plain RuntimeError on timeout if it never resolves to 'ready' or a failed status.

    build_teradata_index ¶

    build_teradata_index(force_rebuild: bool = False) -> None

    Build or refresh the three Teradata vector stores.

    Always reuses an existing collection via add_datasets (which creates the collection on first use, and upserts/appends rows keyed on ID on subsequent calls) rather than destroying and recreating it. destroy() requires admin privileges ("Only admin users can use this method" per Collection.destroy()'s own docstring) and previously ran as a routine step of every force_rebuild=True call — a non-admin session, or any other destroy failure, was silently swallowed, leaving the stale collection in place so the following from_datasets() call then failed with TeradataGenAIException: Collection ... already exists (raised by Collection.create()'s own authoritative existence check). Reusing the collection sidesteps the admin requirement entirely and matches how the Enterprise Vector Store is meant to be refreshed incrementally.

    Known limitation, confirmed live: add_datasets() actually rejects re-adding an object name already attached to the collection ("You can only add object names that do not exist in the collection", TDML_2412 400) — and _from_datasets_kwargs() always names the SAME FS_V_BUSINESS_DICTIONARY_* view, so this call fails on every second-and-later invocation once the source is already attached (i.e. calling build_teradata_index() again after new rows are documented does not currently pick them up via this path). The documented alternative, Collection.update(data_refresh=True) ("refresh/ regenerate vectors ... using existing backing/source tables"), was investigated as a replacement — it succeeded once in isolated testing, then failed consistently on every subsequent attempt (fresh instance, genuinely new unindexed row present) with "At least one of collection_index, collection_parameters, collection_description, or new_collection_name must be provided for update" — an apparently intermittent/stateful server-side issue in teradatagenai, not something reliably reproducible or fixable from tdfs4ds's side. Left as add_datasets() rather than trade a known-broken path for a differently-broken one; a real fix needs a dedicated follow-up once data_refresh's behavior is better understood.

    force_rebuild is now purely informational for logging — both branches behave identically. Note this means rows whose source business-dictionary entry was deleted are NOT purged from the index by this call (no admin-free per-id deletion API is available yet — see Collection.get_ids, which is currently an unimplemented stub upstream); only genuinely new/changed rows are added or refreshed.

    _retry_if_not_ready ¶

    _retry_if_not_ready(fn, name: str, timeout_seconds: float = 120.0, poll_interval: float = 5.0)

    Call fn(), retrying while it raises the transient "not in a ready state" 409 (see _is_not_ready_error) — the collection is still processing a prior async operation (its own creation, or a previous update/add_datasets call). Every Collection write operation is asynchronous ("... request is accepted and in progress"); calling another write immediately after can hit the collection while it's still busy.

    Re-raises the original exception once timeout_seconds elapses, or immediately for any other exception shape (not a not-ready 409).

    _wait_until_destroyed ¶

    _wait_until_destroyed(name: str, timeout_seconds: float = 60.0, poll_interval: float = 2.0) -> bool

    Poll _lookup_store_database(name) until it reports the collection gone, or timeout_seconds elapses.

    Collection.destroy() is asynchronous — it returns as soon as the request is accepted ("Collection destroy request is accepted and in progress"), not once the collection is actually gone. Calling from_datasets() (create) immediately afterward is a race: the collection can still exist server-side, so create collides with TeradataGenAIException: Collection ... already exists even though destroy() itself reported success.

    Returns True if the collection was confirmed gone, False on timeout (caller decides whether that is fatal).

    reset_teradata_store ¶

    reset_teradata_store(sources: Optional[List[str]] = None) -> None

    Destroy and recreate the named Teradata vector store collection(s) from scratch — the true "clean rebuild" that :func:build_teradata_index no longer performs (it always reuses/upserts, see its docstring), so stale rows (business-dictionary entries deleted since the last index) are never purged there.

    Requires the connected user to have admin privileges — destroy() (via :func:_safe_destroy) raises otherwise (see Collection.destroy()'s docstring: "Only admin users can use this method"). Call this explicitly when you know you have admin rights and want stale rows gone; the routine build_vector_index()/rebuild_vector_indexing() path deliberately does not call this, so it keeps working for non-admin sessions.

    Parameters¶

    sources: Subset of "objects", "sections", "columns" to reset. None (default) resets all three.

    _parse_one ¶

    _parse_one(source: str, hit) -> Optional[dict]

    Normalise a single similarity-search hit into the standard row schema.

    search_teradata_index ¶

    search_teradata_index(query: str, sources: list[str] = ('objects', 'sections', 'columns'), section_names: Optional[List[str]] = None, top_k: int = 10) -> pd.DataFrame

    Search the Teradata vector stores using similarity search.

    Cross-store federation (v0.3.1.148): searches the primary store's collections AND, when read-only stores are attached (tdfs4ds.read_stores()), each read-only store's OWN materialized collections — each store keeps its own collections in its own database, so this needs no re-index, only additional queries scoped by (vector_prefix, target_database) per store. Hits are merged, tagged with FEATURE_STORE provenance (only when >1 store), deduped on (OBJECT_NAME, FEATURE_STORE) and run through the shared distance gate. Fail-open: a store whose collections are absent/failed is skipped with a warning (the agent still answers it via the catalog / business-dictionary paths). A single store (the default) is byte-identical to the historical read. Chroma is a single local index and does NOT federate (see search_vector_index).
