    tdfs4ds.agent.chatbot¶

    tdfs4ds.agent.chatbot ¶

    Gradio-based chatbot interface for the tdfs4ds consumer agent.

    Provides a simple web UI that non-technical users can interact with directly from a Jupyter notebook or as a standalone server.

    Usage in a notebook::

    from tdfs4ds.agent.chatbot import launch_chatbot
demo = launch_chatbot()          # opens inline + new-tab link

    Standalone::

    python -m tdfs4ds.agent.chatbot   # starts on port 7860

    consumer_agent(question: str, thread_id: str = 'default', return_trace: bool = False, return_context: bool = False, skills: Optional[List[str]] = None, mcp_enabled: bool = False) -> Union[str, tuple]

    Run the LangGraph agent for a single conversational turn.

    Parameters:

    Name Type Description Default
    question str

    User question in natural language (French or English).

    		 required
    thread_id str

    Conversation thread ID for multi-turn memory.

    		 'default'
    return_trace bool

    When True, includes trace_md in the return tuple.

    		 False
    return_context bool

    When True, includes accumulated_context_graph in the return tuple.

    		 False
    skills Optional[List[str]]

    Optional list of intent names (e.g. ['SEARCH', 'DEFINITION']) to restrict the agent to a subset of skills. When None (default), all installed consumer-agent skills are used. Threads are isolated per skill subset — mixing skills values in the same thread_id is not recommended.

    		 None
    mcp_enabled bool

    When True and tdfs4ds.MCP_SERVER_URL is set, the agent can route questions to the MCP server's tools. The toggle changes the graph variant (different cache key) so it is safe to flip mid-conversation — but note that conversation history is not shared between the MCP-on and MCP-off graph instances.

    		 False

    Returns:

    Type Description
    Union[str, tuple]
    • answer: str when both return_trace and return_context are False (default).
    Union[str, tuple]
    • (answer, trace_md) when return_trace=True and return_context=False.
    Union[str, tuple]
    • (answer, accumulated_context_graph) when return_trace=False and return_context=True.
    Union[str, tuple]
    • (answer, trace_md, accumulated_context_graph) when both are True.

    consumer_agent_stream ¶

    consumer_agent_stream(question: str, thread_id: str = 'default', skills: Optional[List[str]] = None, mcp_enabled: bool = False)

    Generator variant of consumer_agent for real-time step streaming.

    Yields:

    Type Description

    ("step", node_name, partial_state) after each graph node completes,

    where partial_state is the state accumulated so far (all fields set by

    nodes that have already run).

    ("done", answer, trace_md) as the final event with the complete result.

    The MemorySaver checkpointer behaves identically with .stream() as with .invoke() — multi-turn memory is fully preserved.

    build_vector_index ¶

    build_vector_index(force_rebuild: bool = False) -> None

    Build or refresh the three vector-store collections from the business dictionary tables.

    Dispatches to the configured backend (tdfs4ds.VECTOR_STORE_BACKEND): 'chroma' (default) or 'teradata'.

    Parameters:

    Name Type Description Default
    force_rebuild bool

    If True, delete existing collections and re-index from scratch (Chroma backend only — no admin privileges required locally). The Teradata backend does NOT delete on force_rebuild: it always reuses/upserts into the existing collection instead, because deleting there requires admin privileges that may not be available (see vector_store_teradata.build_teradata_index's docstring). Use tdfs4ds.agent.vector_store_teradata.reset_teradata_store() directly for a true destroy-and-recreate on Teradata when you have admin rights and want stale rows purged. If False, upsert all rows (add new, update existing) on both backends.

    		 False

    _build_chat_ui ¶

    _build_chat_ui(initial_thread_id: str, with_skills_pane: bool = True) -> dict

    Build the chat components + wiring inside the current gr.Blocks context.

    launch_chatbot_app ¶

    launch_chatbot_app(port: int = 7860, server_name: str = '0.0.0.0', title: str = 'Feature Store Assistant', description: str = None, inline: bool = False, share: bool = False, thread_id: Optional[str] = None) -> gr.Blocks

    Launch the consumer-agent chatbot with a Teradata-styled tabbed interface.

    The layout uses a left-oriented tab nav (sidebar look) + a hero header, themed via :data:_TD_CSS (Teradata orange/navy palette). Tabs:

    • 💬 Chatbot — the conversational feature-store assistant (left) plus a right Additional skills pane that lists the plugin skills in TDFS4DS_SKILLS_FOLDER and can import/export a skill's SKILL.md.
    • 🔗 Lineage — interactive pyvis dependency graph of the full persisted lineage (FS_LINEAGE_GRAPH). Click Refresh lineage graph to load.
    • 🧪 Evaluation — generate / import / run evaluation scenarios + report.

    • ⚙️ Parameters — live admin controls:

    • Feature store database — switch the active feature store. The target is validated with tdfs4ds.connect(database=..., create_if_missing=False); a database that does not contain a feature store is rejected (no switch).

    • Vector store — backend (chroma/teradata) + Chroma path, and a Refresh/Rebuild button (build_vector_index(force_rebuild=True)).
    • Instruct model / Embedding model — provider, model, URL, API key, proxy (and embedding dimension), applied to the tdfs4ds globals.

    Applying any change that affects the data or the model resets the agent (reset_consumer_agent()), rebuilds the index when needed, rotates the conversation thread, and clears the chat — so the session starts clean.

    The existing :func:launch_chatbot / :func:launch_chatbot_with_index are unchanged; this is an additive wrapper.
