    tdfs4ds.serve.panels¶

    tdfs4ds.serve.panels ¶

    Gradio-free panel logic for the tdfs4ds admin web shell.

    Each function here returns plain JSON-serialisable data or HTML strings — never gradio objects — so they can back FastAPI routes (see :mod:tdfs4ds.serve.app). The heavy lifting (Plotly figures, business-dictionary queries, optimiser, evaluation) is delegated to the already-module-level builder functions in :mod:tdfs4ds.agent.chatbot; this module only orchestrates them and strips the Gradio presentation layer.

    _md_to_html_fragment ¶

    _md_to_html_fragment(md: str) -> str

    Render a Markdown string to a styled, self-contained HTML fragment.

    Uses the markdown package (already a dependency) with the fenced_code, tables and sane_lists extensions so GFM pipe tables, fenced code blocks and lists render properly — unlike the minimal client-side marked() in app.js. Scoped via the .td-md class. Fail-open: returns an escaped <pre> block on any error so the panel never breaks.

    dashboard_report_data(window_days: int = 90) -> dict

    Assemble the plain data dict the inline-SVG report renders from.

    _capture_tdfs4ds_logs ¶

    _capture_tdfs4ds_logs(emit)

    Forward 'tdfs4ds' logger records to emit(str) for the duration.

    The rebuild cores already log per-object lines ("documenting X", "skipped", "absent from graph — fallback", per-seed lineage, per-collection indexing). Capturing them is how the Maintenance panel streams progress WITHOUT adding a progress= param to those cores. Temporarily forces DISPLAY_LOGS=True so the lines actually emit.
