    tdfs4ds.dataset.batch_retrieval ¶

    tdfs4ds.dataset.batch_retrieval¶

    Parser-limit-safe batch retrieval of dataset rows for a list of entity keys.

    A naive batch lookup builds SELECT * FROM ds WHERE (a=1 AND b=2) OR … whose text grows with the number of keys and, for many / wide composite keys, exceeds Teradata's ~1 MB request parser limit (Error 5639). The robust strategy is a volatile-table join: load the (deduplicated) keys into a volatile table whose PRIMARY INDEX matches the actual distribution of the feature-store technical tables the dataset joins, then INNER JOIN — keys travel as data, not SQL text, so there is no parser limit and the join is AMP-local.

    Public API¶

    • :func:retrieve_batch — fetch rows for a list of entity-key tuples, choosing the in_list predicate or the volatile_join strategy (method='auto').
    • :func:resolve_dataset_pi — read the real PRIMARY INDEX / partitioning of the feature tables a dataset joins (so the volatile table co-locates with them).
    • :func:dataset_entity_columns — the dataset grain as [(name, type), …].

    Connection seam¶

    Every helper takes an optional raw teradatasql conn. When conn is None the active teradataml context is used via tdml.execute_sql — so package users get the zero-config path and the benchmark can reuse its own raw per-thread connection (avoiding teradataml's pooling latency).

    resolve_dataset_pi ¶

    resolve_dataset_pi(dataset, entity_cols, conn=None)

    Resolve the REAL primary index / partitioning of a dataset's feature tables.

    Returns a dict::

    {"primary_index": [...], "partitioning": "...",
 "source": "DBC.IndicesV" | "fallback_entity_cols",
 "feature_tables": ["db.FS_T_…", …]}

    The volatile key table should use primary_index so the join co-locates with the feature storage tables. Fail-open: returns entity_cols when the real distribution can't be read (restricted DBC, view-only, no match).

    retrieve_batch ¶

    retrieve_batch(dataset, list_entity_keys, entity_cols=None, *, columns=None, method='auto', volatile_threshold=DEFAULT_VOLATILE_THRESHOLD, conn=None, dedup=True)

    Retrieve rows of dataset for a list of entity-key tuples.

    Parameters¶

    dataset : str Fully-qualified DB.dataset_view to read from. list_entity_keys : list[tuple] Entity-key tuples in the order of entity_cols (each tuple one row's key). entity_cols : list[str] | None Grain/entity key columns. When None they (and their SQL types) are auto-derived from the dataset catalog (needs an active teradataml context). columns : list[str] | None Columns to project; None selects all. method : {'auto','in_list','volatile_join'} 'in_list' builds a WHERE predicate (single-col IN / composite OR-of-ANDs); 'volatile_join' loads the keys into a volatile table co-located on the feature-store primary index and joins; 'auto' (default) uses the volatile join when the unique-key count reaches volatile_threshold OR when the in_list predicate would exceed the parser limit, else the in_list predicate. volatile_threshold : int Key count at/above which 'auto' switches to the volatile join. conn : teradatasql connection | None Optional raw connection (benchmark reuse). None uses the teradataml context. dedup : bool Drop duplicate key tuples first (required for the volatile join; also shrinks the in_list predicate).

    Returns¶

    list[tuple] The fetched rows.
