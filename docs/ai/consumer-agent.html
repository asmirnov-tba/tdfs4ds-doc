    Consumer agent¶

    A conversational chatbot built on LangGraph + Gradio with a pluggable vector store (ChromaDB by default, Teradata Enterprise Vector Store as an opt-in). Business users ask natural-language questions; the agent answers from the business dictionary.

    Architecture¶

    The agent executes a multi-stage LangGraph StateGraph with full cross-turn context persistence via a process-level MemorySaver singleton:

    START
  ↓
classify
  (LLM: intent classification + object name extraction via Pydantic structured output)
  ↓
detect_domain
  (Scan all DATA_DOMAINs; resolve owning domain; persisted across turns)
  ↓
recognize_entity
  (Type-tag the object: feature, dataset, or process view)
  ↓
[conditional routing by intent]
  ├─ SEARCH         → skill_search        → synthesize → END
  ├─ DEFINITION     → skill_definition    → synthesize → END
  ├─ USAGE          → skill_usage         → synthesize → END
  ├─ FRESHNESS      → skill_freshness     → synthesize → END
  ├─ SUMMARY        → skill_summary       → synthesize → END
  ├─ LINEAGE        → skill_lineage       → synthesize → END
  ├─ EXPLAIN        → skill_explain       → synthesize → END
  ├─ ALGORITHM      → skill_algorithm     → synthesize → END (MCP only; dual-intent runs EXPLAIN first)
  ├─ DATASET        → skill_dataset       → synthesize → END
  ├─ BUILD_DATASET  → skill_build_dataset → synthesize → END
  ├─ RECAP          → skill_recap         → synthesize → END (memory only)
  ├─ OUT_OF_SCOPE   → (polite redirect)   → synthesize → END
  └─ DATA_QUERY     → skill_mcp_tools     → synthesize → END (MCP only)

    RECAP and OUT_OF_SCOPE are built-in intents — they have no ca-* SKILL.md file. build_graph() injects their descriptions into the dynamic Pydantic intent model and the classifier prompt, and _build_classifier_examples() seeds explicit few-shot examples so they fire reliably. RECAP reads only the accumulated conversation memory (no Teradata query) to summarise the features, entities, and datasets discussed so far; OUT_OF_SCOPE politely redirects off-topic questions (weather, general coding, etc.).

    Graph Singleton & State Persistence¶

    The graph instance is cached in a module-level dict _graph_instances, keyed by (frozenset(skills), plugin_key, mcp_enabled). This singleton pattern ensures:

    • One MemorySaver checkpoint per (skills, MCP) combo — cross-turn state (resolved_data_domain, resolved_feature_triplet, etc.) persists in LangGraph's built-in checkpointer without an external database.
    • Automatic graph recompilation when skills change or MCP is toggled.
    • Clean reset via reset_consumer_agent() which clears the entire cache.

    Context as a Named-Entity Knowledge Graph¶

    The AgentState TypedDict is a stateful knowledge graph whose fields are persisted across conversation turns. Users can refer back without repeating context:

    State field What it represents
    resolved_data_domain Active DATA_DOMAIN (project namespace)
    resolved_object_name Last explicitly named feature or dataset (remembered)
    resolved_feature_triplet {feature_name, entity_name, process_id, view_name} — the "current node" in the feature graph
    resolved_entity_name Entity type currently in scope (e.g. "CustomerID")
    resolved_feature_list One or many features currently discussed
    resolved_column_sources {COLUMN_UPPER: source_table} — populated by EXPLAIN, enables drill-down DEFINITION
    resolved_dataset_id UUID from the last SUMMARY turn on a dataset
    resolved_dataset_features Features loaded from that dataset
    resolved_entity_name Entity picked during an entity-clarification turn; drives skill_definition's entity hint
    pending_action / pending_action_context State machine for multi-step flows (dataset creation, entity clarification, ambiguous-query clarification)

    The effective current object is resolved by _effective_object(state) with this priority: explicit this-turn → resolved_feature_triplet.feature_name → resolved_object_name.

    Accumulated-context feature keying¶

    accumulated_context_graph.features is keyed by FEATURE_NAME::ENTITY_NAME (both uppercased) — not by the bare feature name — so the same feature name registered under two entities never collides in the cross-turn cache. The features, datasets, entities, and source_views sections all use uppercase keys for stable lookup (processes and freshness_timeline stay keyed by their process-id UUIDs). Skills resolve entries through _feat_cache_lookup(), which tries the exact composite key, then a FEATURE:: prefix scan, then the legacy plain-FEATURE key (so older snapshots still load).

    MCP / DATA_QUERY integration¶

    When tdfs4ds.MCP_SERVER_URL is configured, users can enable the "Enable MCP Tools" checkbox in the chatbot to route questions to remote data sources via the MCP (Model Context Protocol).

    How it works:

    1. The graph registers an additional DATA_QUERY intent and corresponding skill_mcp_tools node.
    2. When MCP is enabled and no feature-store skill matches the question, the classifier may assign DATA_QUERY intent.
    3. _skill_mcp_tools_async() uses langchain_mcp_adapters.MultiServerMCPClient to wrap the remote tools (e.g. SQL query execution, file upload, database operations).
    4. A ReAct agent (langgraph.prebuilt.create_react_agent) chains tool calls until the query is resolved.

    Data governance & safety:

    • _make_guarded_tools() wraps all MCP tools with rejection logic: queries referencing internal feature-store objects (FS_T_*, FS_V_*, process view names) are blocked.
    • SELECT queries automatically get TOP 20 injected; if results hit the limit, a warning is appended so the user knows data was truncated.
    • Event loop compatibility: uses ThreadPoolExecutor inside a running Jupyter loop, asyncio.run() in other contexts.

    Intents¶

    Intent Trigger example What happens
    SEARCH "What features analyse customer spending?" Semantic search over objects / sections / columns collections (Chroma or Teradata Vector Store)
    DEFINITION "What does total_amount measure?" Resolves feature → process view → column doc in FS_BUSINESS_DICTIONARY_COLUMNS
    USAGE "How do I use avg_amount in Tableau?" Audience, granularity, regulatory guidance
    FRESHNESS "Is total_amount up to date?" / "When was DS_CLAIMS last refreshed?" Latest run from follow_up_report(). Works for individual features (resolves single process) and datasets (aggregates across all processes feeding the dataset; returns last_successful_run, days_since_update, has_recent_failure). Returns "not found" only when the named object is neither a registered feature nor a dataset.
    SUMMARY "List all features for customers" / "What features belong to the provider_id entity?" Full feature list with entity + description; scoped to one entity when entity_filter resolves
    LINEAGE "Where does total_amount come from?" build_teradata_dependency_graph upstream
    EXPLAIN "How is total_amount calculated?" SHOW VIEW DDL → LLM plain-language explanation + per-column source map (also accumulates into the context graph)
    ALGORITHM "What algorithm is behind ml_risk_score?" (MCP only) Walks the tdfs4ds→tdstone2 STO metadata chain to explain the ML model: algorithm, hyperparameters, training configuration, Python packages, partition count. Returns answer directly (no second LLM pass). Disabled when MCP is off — algorithm-shaped questions get a "please enable MCP" message instead.
    RECAP "What are the features we are talking about?" Reads accumulated conversation memory only — no Teradata query
    OUT_OF_SCOPE "What's the weather today?" Polite redirect; the question is unrelated to the feature store

    DATASET, BUILD_DATASET, and DATA_QUERY (MCP only) are additional routable intents; see the routing diagram above.

    Compound EXPLAIN + ALGORITHM questions¶

    A compound question "How is ml_risk_score computed? What algorithm is behind it?" carries both EXPLAIN signals ("how is … computed") and ALGORITHM signals ("algorithm"). With MCP on:

    1. Routing — _node_classify_impl re-routes EXPLAIN → ALGORITHM when both signal sets match.
    2. Dual-intent execution — node_skill_algorithm runs skill_explain first (SQL formula), prepends its answer with a --- divider above the ALGORITHM answer (ML model details), and persists the EXPLAIN column sources so a follow-up "what is <column>?" still resolves.

    With MCP off, the algorithm half is unavailable and the question routes to EXPLAIN only (the _looks_like_algorithm_question guard fires the "enable MCP" message in that case).

    Entity-scoped SUMMARY¶

    When the named object resolves to a registered entity (not a feature or dataset), or when entity_filter is supplied, skill_summary lists only the features that belong to that entity — answering "what features belong to entity X?". _recognize_object_type now type-tags entity and data_domain objects in addition to feature / dataset / process.

    State remembered across turns¶

    State field What is remembered
    resolved_data_domain Domain owning the current feature
    resolved_object_name Last explicitly named feature / dataset
    resolved_feature_triplet (feature_name, entity_name, process_id, view_name)
    resolved_entity_name Entity in focus (e.g. CustomerID)
    resolved_feature_list One or many feature names currently discussed
    resolved_column_sources Column → source-table map from the last EXPLAIN turn
    resolved_entity_name Entity picked during an entity-clarification turn

    Follow-up questions that omit the feature name reuse the previous one automatically. When the same feature name exists across multiple entities, the remembered entity silently disambiguates.

    Ambiguous-feature entity clarification¶

    When a feature name exists under several entities (different granularities), the agent first tries to resolve the entity without asking:

    1. Classifier extraction — the intent models carry an entity_name field; "What is total_amount for pharmacy_id?" extracts pharmacy_id directly and it becomes the entity hint for the skill (top priority in the hint chain).
    2. Question scan — _entity_from_question() is a deterministic safety net at every ambiguous_feature site (DEFINITION, FRESHNESS, LINEAGE, EXPLAIN): the raw question is scanned for the candidate entity names (case-insensitive, _/space-normalised, multi-key entities match on any key component) and auto-resolves when exactly one matches.

    Only when neither layer resolves it does skill_definition return error="ambiguous_feature" with the valid entity list. Rather than re-classifying the user's clarifying reply from scratch — which used to spiral into nonsense clarification loops — the agent resolves it deterministically:

    1. node_skill_definition raises a tracked awaiting_feature_entity_clarification pending action (via _ambiguous_feature_pending) carrying the entity list, the original feature, and the origin intent. synthesize_answer renders a numbered menu — a deterministic EN/FR template, not an LLM call (an LLM-rendered clarification was observed embellishing the question with a guessed generic definition of the feature).
    2. On the next turn, _node_classify_impl short-circuits the LLM classifier and resolves the reply through _resolve_clarification_choice (digit / ordinal / fuzzy label).
    3. Role separation + intent inheritance — the entity comes from the reply, while the feature (object_name) and intent (DEFINITION) are inherited from context. So both a bare "1" and a mixed-concept reply like "the total_amount feature of the provider_id entity" resolve correctly. The chosen entity is stored in resolved_entity_name and used as skill_definition's entity hint — the answer comes back scoped, with no second menu.

    Typo recovery¶

    The classifier pins intent to question shape, not name recognition: "what is ml_rsk_scre?" stays DEFINITION even though the token matches nothing (it may be a typo), instead of falling into SEARCH's speculative synthesis. When the documentation lookup then misses, _fuzzy_match_feature (difflib, cutoff 0.75) kicks in: a single close candidate is answered directly with a visible correction note ("'ml_rsk_scre' was not found — answered for the closest match 'ml_risk_score'"); multiple candidates produce a did-you-mean list. DEFINITION and FRESHNESS auto-correct this way, matching the existing LINEAGE / EXPLAIN behaviour.

    Lineage grounding¶

    Lineage answers only ever mention objects that exist in a dependency graph. The cross-turn cache is consulted only when a previous LINEAGE turn actually traced upstream objects — the bare feature → process_view association written by a DEFINITION turn is treated as a cache miss and triggers a real SHOW VIEW/SHOW TABLE walk. The summarisation prompt forbids inventing object names; when no upstream is found the agent says so explicitly instead of calling the LLM.

    Quick start¶

    import tdfs4ds
from tdfs4ds.agent import launch_chatbot_with_index

tdfs4ds.INSTRUCT_MODEL_PROVIDER = 'vllm'
tdfs4ds.INSTRUCT_MODEL_URL      = 'https://api.example.com/v1'
tdfs4ds.INSTRUCT_MODEL_API_KEY  = 'my-key'
tdfs4ds.INSTRUCT_MODEL_MODEL    = 'mistral-7b-instruct'
tdfs4ds.EMBEDDING_MODEL_URL     = 'https://api.example.com/v1/e5'
tdfs4ds.EMBEDDING_MODEL_MODEL   = 'bge-m3'

# One call: build the vector index (incremental) then launch the chatbot
demo = launch_chatbot_with_index(port=7860, force_rebuild=False)

    Programmatic API¶

    from tdfs4ds.agent import consumer_agent

answer = consumer_agent("What features are available?", thread_id="session-1")
answer = consumer_agent("How is nb_days_since_last_transactions calculated?", thread_id="session-1")
answer = consumer_agent("When was it last updated?", thread_id="session-1")
answer = consumer_agent("And for CustomerID?", thread_id="session-1")   # entity remembered, new feature group

    consumer_agent returns just the answer string by default. Two opt-in flags change the return shape:

    answer, trace_md = consumer_agent(q, thread_id="s1", return_trace=True)
answer, trace_md, acc_ctx = consumer_agent(q, thread_id="s1", return_context=True)
    • return_trace=True → (answer, trace_md) — the markdown agent trace.
    • return_context=True → (answer, trace_md, acc_ctx) — a 3-tuple that also returns the accumulated_context_graph dict (the cross-turn knowledge graph), so callers always get the trace alongside the context.

    EXPLAIN → DEFINITION drill-down¶

    After an EXPLAIN turn, the agent lists every variable involved in the formula with its source table. Asking "what is <column>?" resolves the column through FS_BUSINESS_DICTIONARY_COLUMNS — even when the column is not a registered feature.

    USER: How is total_amount calculated?
AGENT: total_amount sums AMOUNT_NET grouped by CUSTOMER_ID.
       Variables involved:
         AMOUNT_NET  ←  SOURCE_DB.TRANSACTIONS
         CUSTOMER_ID ←  SOURCE_DB.TRANSACTIONS

USER: what is AMOUNT_NET?
AGENT: AMOUNT_NET is the net transaction amount in EUR, post-discount …

    Chatbot UI¶

    Agent trace panel¶

    The chatbot includes a collapsible Agent trace accordion showing full execution:

    • Intent Classification — detected intent, object name (this turn vs. remembered)
    • Feature Context — triplet (feature, entity, process ID, view), feature list, column sources
    • Dataset Context — dataset ID and feature count (if applicable)
    • Pending Action — multi-step flow state (e.g. awaiting dataset name)
    • DATA_DOMAIN — available domains, active domain, resolution source
    • Skill Executed — name + input object
    • Skill Result — structured output (for single features) or context graph (for multi-object)
    • Errors include per-step diagnostic messages

    Context knowledge graph panel¶

    A second collapsible accordion displays an interactive Plotly network graph visualising the agent's accumulated context:

    • Nodes (colour-coded by type):
    • Domain (blue diamond)
    • Entity (green square)
    • Features (orange circles)
    • Process views (grey squares)
    • Datasets (purple triangles)
    • Column sources (red nodes, when EXPLAIN was used)
    • Edges show relationships: domain → entity → features → views → datasets

    The graph is rebuilt after each turn to reflect new resolutions, multi-object DEFINITION expansions, and EXPLAIN column mappings.

    Admin parameters panel (launch_chatbot_app)¶

    launch_chatbot_app() is an additive wrapper around the chatbot: same chat UI, plus a collapsible ⚙️ Parameters panel for changing configuration live — without restarting the process. (launch_chatbot and launch_chatbot_with_index are unchanged; use them when you don't need the panel.)

    from tdfs4ds.agent import launch_chatbot_app

# Same prerequisites as launch_chatbot (a connected context + built index).
demo = launch_chatbot_app(port=7860)

    The panel edits the same globals as the agent

    The admin panel does not keep its own configuration. Every field maps directly to the same tdfs4ds.* global that the consumer agent already reads (tdfs4ds.SCHEMA, INSTRUCT_MODEL_*, EMBEDDING_MODEL_*, VECTOR_STORE_BACKEND, CHROMA_PATH, …). Applying a change in the panel is exactly equivalent to setting that global in Python, so it also affects any consumer_agent(...) / query_optimizer_agent(...) calls made from the same process afterwards. Values are pre-filled from the current globals when the app starts — including whatever was loaded from env vars / .env / JSON config (see Configuration).

    The panel has four groups; each has an Apply button and a status line:

    Group Fields What Apply does
    Feature store database database name tdfs4ds.connect(database=…, create_if_missing=False). Validated: a database that does not contain a tdfs4ds feature store is rejected and the active store is not changed. On success: sets tdfs4ds.SCHEMA, rebuilds the index, resets the agent, clears the chat.
    Vector store backend (chroma/teradata), Chroma path; Refresh / rebuild button Sets VECTOR_STORE_BACKEND / CHROMA_PATH and rebuilds. Refresh alone re-runs build_vector_index(force_rebuild=True) — use it after the business dictionary changes, since the local Chroma index does not auto-update.
    Instruct model provider, model, URL, API key, proxy Sets INSTRUCT_MODEL_* (incl. INSTRUCT_MODEL_PROXY) and resets the agent. The next LLM call builds a fresh client from these globals.
    Embedding model provider, model, URL, API key, proxy, dimension Sets EMBEDDING_MODEL_*, then rebuilds the index (embeddings changed) and resets the agent.

    Any change that affects the data or the model rebuilds the index when needed, resets the agent (reset_consumer_agent()), rotates the conversation thread, and clears the chat so the session starts clean against the new configuration.

    When run via the API server (python -m tdfs4ds.serve), this admin app is served automatically on a second port — see Deployment → OpenAI-compatible API server.

    Database choice is process-wide — isolate per user with separate processes

    Because the panel mutates process-wide globals, switching the feature-store database changes it for everyone served by that process. To give each user their own database, run a separate process / container per user (one container per user, each with its own TDFS4DS_SCHEMA) rather than sharing one — see Per-user database choice. There is no per-session database isolation within a single process.

    Business dictionary & retrieval architecture¶

    The agent answers from a 3-level documentation hierarchy designed for chunk-free hierarchical RAG, using both structured SQL and vector search.

    Documentation tables (all VALIDTIME temporal)¶

    Level Table What it stores Populated by
    0 FS_BUSINESS_DICTIONARY_OBJECTS Object overview (BUSINESS_DESCRIPTION) document_process, document_dataset_incremental
    1 FS_BUSINESS_DICTIONARY_SECTIONS Named sections per object (OVERVIEW, ENTITY, FEATURE_THEMES, BUSINESS_QUESTIONS, INTENDED_AUDIENCE) document_dataset_incremental
    2 FS_BUSINESS_DICTIONARY_COLUMNS Per-column descriptions (feature name → view name → description) document_process, document_dataset_incremental

    What gets vectorised¶

    The vector store maintains 3 collections, populated by build_vector_index():

    • {prefix}_objects ← BUSINESS_DESCRIPTION from OBJECTS
    • {prefix}_sections ← SECTION_CONTENT from SECTIONS
    • {prefix}_columns ← BUSINESS_DESCRIPTION from COLUMNS

    The prefix is tdfs4ds for Chroma (cosine-distance HNSW collections) and tdfs4ds.TERADATA_VECTOR_STORE_PREFIX for the Teradata backend.

    Documents are embedded in batches of 8 to respect rate limits (OpenAI, TEI, etc.).

    Backend selection¶

    tdfs4ds.VECTOR_STORE_BACKEND = 'chroma'    # default — local or remote ChromaDB
# or
tdfs4ds.VECTOR_STORE_BACKEND = 'teradata'  # Teradata Enterprise Vector Store

    build_vector_index() and search_vector_index() dispatch on this setting; all skill code remains unchanged. The Teradata backend persists the three collections as named TeradataVectorStores directly in Vantage and accepts either a LangChain Embeddings object (default — keeps your vLLM / TEI / OpenAI-compatible setup) or a teradatagenai.TeradataAI object (set TERADATA_AI_API_TYPE to opt in). See LLM & embedding models for the full configuration.

    Hybrid retrieval strategy¶

    Each skill uses a mix of structured SQL (for precision) and vector search (for recall when names don't match exactly):

    Skill Structured lookups Vector search
    SEARCH BD_SECTIONS (FEATURE_THEMES, BUSINESS_QUESTIONS), BD_OBJECTS, catalog Yes — semantic priority, results prepended
    DEFINITION Feature catalog → process view → BD_COLUMNS; BD_OBJECTS + BD_SECTIONS for datasets Fallback only (when SQL returns nothing)
    USAGE BD_SECTIONS (INTENDED_AUDIENCE, OVERVIEW) Fallback only
    FRESHNESS Feature catalog → process catalog → follow_up_report (single feature); dataset catalog → all process_ids for the dataset's features → aggregated follow_up_report (dataset) None
    SUMMARY Feature catalog + BD_SECTIONS (FEATURE_THEMES, OVERVIEW) None
    LINEAGE build_teradata_dependency_graph (DDL walk) None
    EXPLAIN SHOW VIEW DDL None

    For multi-object DEFINITION queries, the skill also builds a context_graph dict (features, shared views, shared datasets) stored in skill_result and visualised in the chatbot's context panel. EXPLAIN does the same — _skill_explain_multi returns a context_graph (carrying each feature's process_id + view_name) and a resolved_triplet focus pointer, which node_skill_explain merges into accumulated_context_graph, so a follow-up turn after an EXPLAIN can resolve from cache.

    Vector index management¶

    from tdfs4ds.agent import build_vector_index, search_vector_index

build_vector_index(force_rebuild=False)    # incremental: indexes only new rows (Chroma)
build_vector_index(force_rebuild=True)     # full rebuild: wipe collections and reindex
search_vector_index('total_amount', k=5)   # raw vector hits (not used directly by skills)
