    Changelog¶

    All notable changes to tdfs4ds are documented in this file.

    [0.3.1.148] - 2026-07-14¶

    Added — cross-store read federation (read-only extended connections)¶

    • tdfs4ds.SCHEMA_READONLY (default []) + extend_read_stores(databases) / detach_read_stores(databases=None) / list_read_stores() and connect(..., databases_readonly=[...]) (also TDFS4DS_SCHEMA_READONLY env / JSON schema_readonly) — connect to a primary store read-write as usual and extend the session to one or more other feature stores read-only. Each store is a string ('DB_B') or a dict descriptor {'schema','schema_view','vector_prefix'}.
    • Reads federate, writes don't. When read-only stores are attached, catalog / business-dictionary / lineage reads become a UNION ALL across the primary + read-only stores (each row tagged with a synthetic FEATURE_STORE provenance column, primary first), and build_dataset can pull a read-only store's features into a dataset built in the primary. upload_features / documentation / lineage writes still target only tdfs4ds.SCHEMA.
    • Byte-identical default. SCHEMA_READONLY == [] ⇒ every read funnel keeps its historical single-store form (the union builders return None) — no behavioural change unless a read-only store is attached. Same opt-in shape as the database_view split.
    • Teradata vector search federates live — search_teradata_index searches each store's own materialized collections by (vector_prefix, target_database) and merges hits (no re-index; a store without collections is skipped fail-open, still answerable via catalog/BD paths). Chroma (single local index) does not federate — out of scope.
    • New module tdfs4ds/read_federation.py (attach/validate + the shared catalog_union_sql / federated_* builders) and accessors read_view_schemas() / read_table_schemas() / read_stores(). Attach validation rejects a version-skewed / ungranted store (fail-open); optional has_bd/has_lineage capabilities are probed so a store without docs or lineage never black-holes those unions.
    • Documented collision limit: the consumer agent's identity/counting/domain-resolution is store-blind — exact for stores with disjoint domains/entities/feature names (the common case); a same-(domain, entity, feature)-across-stores clash resolves to the primary. Full provenance-in-identity hardening is a named follow-up.
    • Offline unit tests: tests/test_read_federation.py.

    [0.3.1.147] - 2026-07-13¶

    Added — parallel-execution planner, FS_TIME_MANAGER, "Execution plan" admin panel¶

    • tdfs4ds.analyze_parallel_plan(seeds=None, all_domains=True) — seed a list of processes / datasets / features, resolve each to its producing process(es), expand to the upstream process closure (everything needed to (re)build them — a self-contained migration unit), and group the closure into dependency waves: a topological layering of the process DAG (derived from the persisted FS_LINEAGE_GRAPH) where every process in a wave is mutually independent and can run in parallel, and waves run in order. seeds=None plans the whole store. The result also reports per-wave manager groups, unify_candidates (equivalent but distinct FilterManager/TimeManager objects), a serialize_together write-conflict guard, and processes with unknown lineage.
    • tdfs4ds.generate_execution_script(plan, mode='process_pool', ...) / write_execution_script(plan) — turn a plan into a standalone .py that runs waves sequentially and processes within a wave in parallel via separate OS processes (each worker opens its own Teradata context from TDFS4DS_TD_* env — safe against the module-global and shared-connection-pool hazards that make in-process threading unsafe).
    • tdfs4ds.unify_filter_managers / unify_time_managers (opt-in, dry_run=True by default) — repoint parallel-candidate processes onto one canonical manager (rewrite the process view's manager reference + update the catalog row).
    • New FS_TIME_MANAGER catalog — a per-process TimeManager association mirroring FS_FILTER_MANAGER, exposed as TIME_DATABASE_NAME/TIME_VIEW_NAME/TIME_TABLE_NAME on FS_V_PROCESS_CATALOG. It is populated by roll_out — as roll_out(process_list, time_manager, …) runs it records each process's TimeManager (idempotent, non-fatal). A TimeManager is a roll-out driver, not a registration-time property, so upload_features is unchanged (no timemanager= argument; registration SQL is byte-identical). Fully backward compatible and self-populating: connect() creates the (empty) table on pre-existing installs before refreshing the catalog view, and simply running your existing rollout fills it in — no migration step, including for processes registered by older versions.
    • Admin "Execution plan" panel — pick datasets/processes, see an interactive wave-layered workflow graph (what runs in parallel vs sequentially), and download the generated .py.

    [0.3.1.146] - 2026-07-13¶

    Added — database_view: optional separate database for views vs tables¶

    • New opt-in configuration for environments that keep views in one database and tables in another (governance / access-control policy). setup(database, database_view=...) and connect(database=..., database_view=...) accept a second database dedicated to VIEW objects, exposed as the global tdfs4ds.SCHEMA_VIEW (also settable via the TDFS4DS_SCHEMA_VIEW environment variable / .env, or the schema_view JSON config key — same tier as SCHEMA).
    • Fully backward compatible. database_view=None (the default) makes SCHEMA_VIEW resolve to SCHEMA through the new tdfs4ds.view_schema() accessor (SCHEMA_VIEW or SCHEMA), so every generated statement is byte-identical to before — no behavioural change unless a view database is explicitly configured.
    • When set: every physical TABLE stays in SCHEMA (feature FS_T_* tables, all catalog/_HIDDEN, business-dictionary, and lineage base tables) and every VIEW — process views, dataset views, all framework FS_V_* public views, per-feature value views, and the business-dictionary / lineage / process-catalog / dataset-catalog views — is created and read in SCHEMA_VIEW. Implementation: table sites keep tdfs4ds.SCHEMA; view create/read sites resolve through tdfs4ds.view_schema(). A view's FROM that reads a table keeps SCHEMA (tables never move); only a view-over-view FROM resolves to the view database.
    • Unqualified names now default to the view database. Registering a process/dataset with an unqualified name (e.g. upload_features(..., target_view_name='my_view')) places the view in SCHEMA_VIEW automatically (relaxing the previous hard "must be schema-qualified" rejection); an explicit db.view is still honoured. Covers _resolve_feature_source, build_dataset, and get_process_id.
    • FEATURE_DATABASE (feature catalog) stays the table database so cross-database dataset views resolve their "{FEATURE_DATABASE}"."{FEATURE_TABLE}" sources; DATASET_DATABASE records the dataset view database. DatasetCatalog gained schema_name_view / existing_view_objects.
    • Operational precondition (not enforced in code): the view database must be granted SELECT (and WITH GRANT OPTION to expose views to downstream consumers) on the table database.
    • Migration for an existing single-database store: set database_view and call connect(create_if_missing=True) to (re)create the framework views in the new database. Stale views left behind in the old database are not dropped automatically.
    • Out of scope (unchanged): TimeManager / FilterManager backing objects (placed by their own explicit schema_name) and the Enterprise Vector Store collection's own target_database.

    [0.3.1.145] - 2026-07-13¶

    Fixed — openai-compatible provider now uses response_format, not vLLM guided_json (LiteLLM / Mistral gateways)¶

    • Bug: documenting/scoring a SQL query through an OpenAI-compatible gateway that fronts a non-vLLM backend (e.g. a LiteLLM proxy → Mistral) failed with UnprocessableEntityError: 422 … "loc":["body","guided_json"], "msg":"Extra inputs are not permitted". Root cause: the openai-compatible (and vllm) provider path sent vLLM's guided_json structured-output parameter — a vLLM-only extension, not part of the OpenAI API — which such gateways reject. A provider literally named "openai-compatible" must speak the OpenAI API, i.e. use response_format.
    • Fix: openai-compatible is reclassified into the response_format json_schema group (alongside openai/azure/mistralai; strict is set only for openai/azure, which Mistral and generic gateways reject). vllm is unchanged — a real vLLM server still gets guided_json. Use provider vllm for a raw vLLM server, openai-compatible for any OpenAI-protocol gateway.
    • Centralized the provider → structured-output decision (mechanism and schema wrapping) into a single source of truth, _build_provider_llm_caller(llm, provider, base_schema, schema_name) in genai/documentation.py. The scattered per-site provider branching (build_documentation_json_schema, build_sql_documentation_chain, build_explain_documentation_chain, and the inline dataset path) — the reason the branches had drifted — now all route through it.
    • One-shot fallback: if a backend rejects the structured-output parameter (_is_structured_output_rejection — matches the 4xx extra_forbidden / "Extra inputs are not permitted" / "unknown|unsupported field/parameter" shapes naming guided_json / response_format / format), the caller retries once unconstrained and relies on the fence-tolerant _robust_json_parser. A genuine model error (context length, rate limit, content policy) never names those parameters, so it is not swallowed. This makes any gateway work even if it supports neither mechanism, so this class of bug cannot recur.
    • No config change required for existing users; no behaviour change for vllm, openai, azure, bedrock, gcp, litellm, or ollama. Immediate workaround for anyone already hitting this on a Mistral-backed gateway: set INSTRUCT_MODEL_PROVIDER='mistralai' (or now 'openai-compatible').
    • Tests: tests/test_structured_output_provider.py (pure unit, no Teradata/network) — mechanism selection per provider, the rejection detector against the real ticket message, and the one-shot fallback (incl. that unrelated errors are not swallowed).

    [0.3.1.144] - 2026-07-12¶

    Added — OFS backing-table storage for TimeManager / FilterManager (Vantage Cloud Lake + tdstone2 Apply)¶

    • Opt-in storage= constructor kwarg on both TimeManager and FilterManager (time_management.py, filter_management.py): None (default) = system default (block storage), 'OFS' = Object File System (MAP = TD_MAP2, STORAGE = TD_OFSSTORAGE). A companion ofs_map= kwarg (default 'TD_MAP2') overrides the map. The choice is per manager, never a global — a user can run block-storage (full primary cluster) processes and OFS/Apply processes side by side in the same session.
    • Context: on VCL, a tdstone2 model scored via the Apply backend runs on the Analytical Compute Cluster (ACC), which reads from OFS. storage='OFS' lets a manager's backing table live on OFS instead of block storage.
    • Making a manager-referencing view score on the ACC needs BOTH (confirmed live on VCL2): (1) the manager table on OFS (storage='OFS') so the ACC can read it — a block-storage read raises Error 6881; and (2) the filter written as a JOIN to the manager view, not a correlated scalar subquery — the ACC rejects ... WHERE d <= (SELECT BUSINESS_DATE FROM <tm_view>) with 6881 but accepts a JOIN. Verified: OFS + JOIN scores (~114s); OFS + scalar subquery still 6881s. So storage='OFS' is necessary but not sufficient on its own — it is genuinely part of the working recipe (the ACC reads the manager table through the JOIN), and the JOIN filter shape is the user's responsibility (a property of the scoring view's SQL). A JOIN keeps the reference dynamic (time-travel via update() still works); a hard-coded literal also passes but freezes the date. No literal-inlining helper or roll_out integration is added — the query shape is authored by the user.
    • roll_out/run() still cannot drive a tdstone2-Apply model — its ingestion CTAS (CREATE VOLATILE TABLE AS SELECT * FROM <live scoring view>) also hits 6881; call model.score() first, then upload_features.
    • New helper module utils/storage.py: resolve_storage, storage_clause(storage, ofs_map) (returns the CREATE TABLE options fragment — '' for the default so DDL is byte-identical to before, or , MAP = …, STORAGE = TD_OFSSTORAGE for OFS), is_acc_ineligible_error, and ACC_6881_HINT. The clause is injected immediately after the table name in the 5 backing-table CREATE sites (TimeManager load_time_steps / clone_timer hard / take_ownership; FilterManager load_filter / clone_filter hard).
    • Clearer 6881: _compute_features_batch (execution.py) now logs ACC_6881_HINT (still re-raises the original) when its own ingestion CTAS hits a 6881.
    • Docs: the OFS + JOIN Apply pattern (and the score()-then-upload_features sequencing) is documented across the time-manager / filter-manager user guides, the fs-rollout / fs-filter skills, and the tdstone2-troubleshooting 6881 table. No score-hook / query rewriter is coded — the filter shape is the user's responsibility (guidance only).
    • Backward compatibility: default behaviour unchanged (no MAP/STORAGE clause). New kwargs are keyword-only with defaults; existing positional calls and run()'s read-only FilterManager(...) reconstruction are unaffected. Offline tests in tests/test_manager_storage.py.

    [0.3.1.143] - 2026-07-10¶

    Added — 'uaf art view' process type (UAF features whose value lives in a secondary ART layer)¶

    • New opt-in, additive PROCESS_TYPE 'uaf art view' for UAF (Unbounded Array Framework) features whose value lives in a secondary ART (Analytical Result Table) layer — ARIMA fit-metrics (R_SQUARE/MAPE/AIC in ARTFITMETADATA), residual tests over ARTFITRESIDUALS, TD_PACF, etc. Reaching such a layer needs two statements (materialise the ART, then TD_EXTRACT_RESULTS over a LAYER), which the single ingestion CTAS (CREATE VOLATILE TABLE AS (SELECT * FROM <view>)) cannot express — so the inline-EXECUTE FUNCTION-in-a-view trick that serves inline-output operators (TD_ACF, TD_DICKEY_FULLER) does not extend to it.
    • upload_features(..., art_sql=[...], art_layers=[...]) — new kwargs. art_sql is a list of EXECUTE FUNCTION ... INTO VOLATILE ART(<name>) statements; art_layers names the declared LAYER(...)s the feature view reads. df/sql_query+target_view_name still build the plain feature view (reading the layer via TD_EXTRACT_RESULTS(ART_SPEC(..., LAYER(...)))).
    • Storage: no DDL migration. art_sql/art_layers are stored in the existing METADATA JSON(32000) column of the process catalog (a JSON array preserves the execution order of art_sql, which is load-bearing — a later statement may read an ART an earlier one created). PROCESS_TYPE is parameterised via kwargs.get('process_type', 'denormalized view') in the registration builders, so existing callers produce identical SQL.
    • run(): a new elif process_type == 'uaf art view' branch reads art_sql/art_layers back from METADATA and threads art_sql through _upload_features → _compute_features_batch, which — per FilterManager partition, after filtermanager.update(), before the ingestion CTAS, in the same session — calls the new _materialise_art(art_sql): for each statement in list order, DROP TABLE <art> (idempotent re-run) then execute it. The session-scoped volatile ART is then visible to the CTAS reading the feature view. Partition-awareness comes from the store filter living in the series-input view's scalar subquery (same pattern as FilterManager/fs-filter).
    • AI-feature gates: optimize_query is force-disabled for this type (art_sql is an opaque UAF invocation and the view EXPLAIN needs the ART materialised first); document_process_incremental skips the view-EXPLAIN and feeds the art_sql operator + art_layers into the column-documentation LLM context; get_process_info additively surfaces ART_SQL/ART_LAYERS.
    • Lineage traces through the ephemeral ART: _extract_series_art_table_names (lineage/lineage.py) parses TABLE_NAME(...) out of SERIES_SPEC/ART_SPEC (inert on non-UAF SQL), and record_process_lineage(art_sql=, art_layers=) rewrites the dangling feature_view → <ephemeral ART> edge into a two-hop path feature_view → <operator/layer node> → series_input_source (fail-open).
    • Backward-compatible: art_sql=None takes the classic 'denormalized view' path byte-for-byte; the 'denormalized view'/'tdstone2 view' branches are untouched.
    • Live-verified end-to-end on Vantage Cloud Lake (Vantage 20): ARIMA fit-metrics R_SQUARE/MAPE from ARTFITMETADATA, 2-partition FilterManager — registration, METADATA, both-partition ingestion (values matching the ART ground truth), idempotent run(force_compute=True), and lineage-through-ART all pass. See the notebook under notebook dev/uaf art view/. Creation-time lifecycle constraint (verified): a view referencing a volatile ART cannot be REPLACE VIEW-created unless the ART already exists in the session (Error 3807) — materialise the ART once before creating the feature view; run() re-materialises it per partition thereafter.

    Added — fs-tdstone2 skill¶

    • New Claude Code skill fs-tdstone2 (a tdfs4ds skill) documenting the one-call bridge tdfs4ds.upload_tdstone2_scores(model) — registering a trained tdstone2 model's per-partition predictions / computed features as governed features (entity derived from the model's mapper; snapshot vs recomputable-view trade-off; the ML-model lineage the ca-algorithm skill walks backwards).

    Tests¶

    • tests/test_uaf_art_view.py — DB-free unit tests for the 'uaf art view' helpers (_extract_art_name, _extract_series_art_table_names, _uaf_art_rewrite).

    [0.3.1.140] - 2026-07-09¶

    Fixed (Teradata Enterprise Vector Store — _inputdata view permanently referencing a garbage-collected ephemeral view)¶

    • Symptom: on a fresh VCL2 test deployment, build_teradata_index() created all three collections without error and collection_tdfs4ds_<source>_index (the embeddings table) held real, non-empty rows (13/16/73 for objects/sections/columns) — but every similarity_search call, and even a plain SELECT COUNT(*) against collectionV_tdfs4ds_<source> from a completely unrelated client (a teradata-mcp5 MCP session, a fresh teradataml session), failed identically with Error 3807 / TDML_2412: Object 'data_scientist.ml___frmqry_v_<id>' does not exist — every time, not transiently (3/3 retries failed identically; the referenced object was confirmed absent from DBC.TablesV).
    • Root cause, found by inspecting the actual DDL of the backing Teradata objects: collection_tdfs4ds_<source>_inputdata is itself a view, and its stored SHOW VIEW DDL hardcoded FROM "data_scientist"."ml___frmqry_v_<id>" — a short-lived materialized-query view that Vantage garbage-collects almost immediately. collectionV_tdfs4ds_<source> (the join view search reads through) depends on _inputdata, so it inherited the same permanent breakage. Traced this to tdfs4ds's own call site (not just the teradatagenai client library, as the 0.3.1.131 fix assumed): _from_datasets_kwargs() built its data= argument via tdml.DataFrame.from_query("SELECT ... FROM <permanent view>") — and confirmed live that DataFrame.from_query(...) itself sets _table_name to a freshly materialized ephemeral view at construction time, before teradatagenai is even called. teradatagenai then bakes whatever _table_name it receives permanently into _inputdata's DDL — so reading from a permanent source view (the 0.3.1.131 fix) was necessary but not sufficient; the from_query() wrapper itself was creating a doomed ephemeral view on every call.
    • Fix (agent/vector_store_teradata.py): _from_datasets_kwargs() now passes data= as a plain fully-qualified "schema.view" string naming the permanent FS_V_BUSINESS_DICTIONARY_* view directly — never a from_query() result — guaranteeing no ephemeral materialization ever happens. Matches teradatagenai's own documented simplest input shape (its from_datasets docstring's own Example 1 passes a plain table name string).
    • No row-level filtering of data= — by design, not an oversight: checked directly that neither from_datasets nor add_datasets exposes any filter/predicate/where-clause parameter. Considered and rejected pointing data= at a filtered subset for scoped/incremental reindexing: _inputdata is a live view (re-evaluated on every query, not a frozen snapshot), so a filtered data= would permanently narrow what the whole collection can ever return. Separately, _index's embeddings are keyed by TD_ID, which _inputdata (re-)assigns via ROW_NUMBER() OVER (ORDER BY <text column>) on every evaluation — a stable-looking key that is actually recomputed by row position, not a persistent identity. Always pass the entire, unfiltered source view.
    • Required recovery step for existing installs: collections created before this fix have the broken ephemeral-referencing _inputdata/collectionV_* DDL baked in permanently — an incremental add_datasets on top will NOT repair it. Call tdfs4ds.agent.vector_store_teradata.reset_teradata_store() (requires admin) once to destroy and recreate the collections using the fixed code path.

    Fixed (Teradata Enterprise Vector Store — search_teradata_index silently returning 0 hits on genuine matches)¶

    • Symptom: after fixing the _inputdata/ephemeral-view bug above and confirming similarity_search() genuinely returns real, relevant hits when called directly, tdfs4ds.search_vector_index()/search_teradata_index() — the actual public entry point — still returned 0 rows for the same query against the same collection, with no error or warning logged.
    • Root cause: _parse_one()'s _TEXT_KEYS (("content", "page_content", "text", "document")) assumed a generic/Chroma-like hit shape. A real Teradata Vector Store hit is flattened differently: the text lives directly under the source's actual business-dictionary column name — business_description (objects/columns) or section_content (sections) — not any of the generic keys, and metadata (object_name, section_name, column_name, etc.) is flattened into the same dict rather than nested under a metadata key. Every hit's text therefore stayed empty, silently dropped by the if not text.strip(): return None guard — 5 genuine hits parsed down to 0 rows with no diagnostic trace.
    • Fix (agent/vector_store_teradata.py): added "business_description" and "section_content" to _TEXT_KEYS. Verified live: the same query that previously returned 0 rows through search_vector_index() now returns 8 real, relevant hits across objects and sections, matching what similarity_search() returns directly.

    Known limitation (Teradata Enterprise Vector Store — build_teradata_index's reuse branch cannot pick up new rows via add_datasets; data_refresh=True investigated but found unreliable)¶

    • Symptom: calling build_teradata_index() a second time (collection already exists, e.g. after documenting one more table/column) fails with TDML_2412 ... Response Code: 400, Message: You can only add object names that do not exist in the collection — and this 400 is then misrouted through _reraise_if_service_down's generic "response code:" catch and re-raised as the unrelated _SERVICE_DOWN_MSG ("service is unreachable or not enabled"), even though the service is clearly up (the same collection had just been created moments earlier).
    • Root cause: add_datasets()'s real semantics are "attach a new source object not yet registered on the collection," not "resync/upsert rows from an already-attached source." _from_datasets_kwargs() always names the SAME FS_V_BUSINESS_DICTIONARY_* view on every call, so any second call through the reuse branch is guaranteed to hit this 400 once the source is already attached — i.e. new business-dictionary rows are NOT currently picked up by a second build_teradata_index() call.
    • Investigated and rejected as a fix: Collection.update(data_refresh=True) — the documented mechanism ("refresh/regenerate vectors for the collection using existing backing/source tables," per its own docstring's Example 8) — looked like the correct replacement and succeeded once in isolated live testing (a new business-dictionary row correctly appeared in _index/_inputdata/ collectionV_*, all in lockstep, 74/74/74). However it then failed consistently on every subsequent attempt — with a fresh TeradataVectorStore instance, and even with a genuinely new, never-indexed row present in the source view — with "At least one of collection_index, collection_parameters, collection_description, or new_collection_name must be provided for update". This looks like an intermittent/stateful server-side issue in teradatagenai, not something reproducibly fixable from the client side, so it was not adopted as the fix — shipping it would trade one known-broken path for a differently unreliable one. build_teradata_index's reuse branch remains on add_datasets() (pre-existing behavior, known limitation). A real fix needs a dedicated follow-up once data_refresh's actual failure conditions are understood.

    Fixed (Teradata Enterprise Vector Store — add_datasets() silently corrupting collections; "no exception" ≠ "write succeeded")¶
